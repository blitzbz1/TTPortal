# Data Model: User Authentication

**Branch**: `001-user-auth` | **Date**: 2026-03-26

---

## Entities

### User

Represents an authenticated person on TT Portal.

| Field | Type | Constraints | Notes |
|-------|------|-------------|-------|
| id | UUID (PK) | Auto-generated by auth provider | Supabase Auth UID |
| full_name | text | Required, non-empty | Display name |
| email | text (unique) | Required, valid email format | Primary identifier; may be Apple relay address |
| avatar_url | text | Nullable | Profile photo URL (future use) |
| city | text | Nullable | Home city (future use) |
| lang | enum (`ro`, `en`) | Default: `ro` | UI language preference |
| auth_provider | enum (`email`, `google`, `apple`) | Required | Primary auth method used |
| created_at | timestamp | Auto-set on creation | |

**Relationships**:
- One User → many Reviews (future: `reviews.user_id`)
- One User → many Venues (future: `venues.created_by`)
- One User → many Check-ins (planned)
- One User → many Friendships (planned)

**Identity & Uniqueness**:
- Email is the unique identity anchor across all auth methods
- When a user signs in via Google/Apple with an email matching an existing email/password account, the accounts are linked (single user record)
- Apple "Hide My Email" relay addresses are stored as-is; they function as normal email addresses

**State transitions**: None for this feature. Users are immediately active upon registration (no email verification, no approval queue).

---

### Session

Managed by the auth provider. Not stored in a custom table — handled via auth provider tokens stored in `expo-sqlite` on-device.

| Aspect | Detail |
|--------|--------|
| Creation | On successful login/registration/OAuth |
| Persistence | Stored on-device via `expo-sqlite`; survives app close |
| Duration | Minimum 7 days of inactivity (per SC-008) |
| Refresh | Auto-refreshed by auth provider SDK |
| Termination | On explicit logout (clears stored tokens) |

---

### Password Reset Token

Managed by the auth provider's built-in password reset flow. Not stored in a custom table.

| Aspect | Detail |
|--------|--------|
| Trigger | User requests reset via forgot password screen |
| Delivery | Email with magic link |
| Expiry | 60 minutes from creation |
| Single-use | Token invalidated after first use |
| Enumeration protection | Same response shown regardless of whether email exists |

## Existing Tables (no changes)

These tables are already in production and require **no schema changes** for the auth feature:

| Table | Auth impact |
|-------|-------------|
| `cities` | None — read-only, public |
| `venues` | Future: add `created_by` (UUID FK → users) when attributing venue submissions to users. Not in scope for auth feature. |
| `reviews` | Future: add `user_id` (UUID FK → users) when attributing reviews to users. Not in scope for auth feature. |

**Note**: The auth feature gates write access (add venue, write review, edit venue) at the **application layer** via route protection. Database-level Row Level Security (RLS) policies will be added in a future iteration to enforce this at the data layer.

| User | password | Minimum 8 characters (email auth only) |

## Auth Provider Configuration

The `users` table above maps to Supabase Auth's `auth.users` system table plus a public `profiles` table (or `users` table in the public schema) for app-specific fields. The auth provider manages:

- Password hashing and verification
- OAuth token exchange (Google, Apple)
- Session token issuance and refresh
- Password reset token generation and validation
- Rate limiting on failed login attempts
